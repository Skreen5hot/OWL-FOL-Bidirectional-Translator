# v3.3 Change Summary (Final)

This document explains what's in v3.3 of the relations catalogue, why those changes were made, and what Aaron needs to do to finalize it.

## What v3.3 is

v3.3 is the relations catalogue regenerated from authoritative sources rather than authored from prior catalogue versions. Every CCO row was verified against the loaded BFO 2020 + CCO + IAO triplestore via SPARQL queries before being committed to the file. Every BFO row was verified by a direct query against loaded BFO 2020. The deterministic module-assignment rule (namespace + property-path closure to BFO ancestor) was applied programmatically; deviations from the rule are explicitly marked with provenance tags.

The catalogue contains 135 relation rows organized in 9 modules:

- core/bfo-2020 — 40 rows
- core/iao-information — 4 rows
- cco/realizable-holding — 20 rows
- cco/mereotopology — 15 rows
- cco/measurement — 8 rows (NEW per v3.3 ADR)
- cco/aggregate — 5 rows (NEW per v3.3 ADR)
- cco/organizational — 14 rows (NEW per v3.3 ADR)
- cco/deontic — 8 rows (NEW per v3.3 ADR)
- ofi/deontic — 8 rows
- (no module — meta-level / W3C standard) — 13 rows

## What's new vs v3.2

### Eight fixes to silently-wrong content in v3.2

These were caught by querying the source ontologies directly:

1. **Denotes IRI corrected**: v3.2 had `cco:ont00001052`. This IRI does not exist in the loaded triplestore. Correct IRI is `obo:IAO_0000219`.

2. **Designates IRI corrected**: v3.2 had `cco:ont00001058`. This IRI does not exist in the loaded triplestore. Correct IRI is `cco:ont00001916`.

3. **Has Process Part module corrected**: v3.2 placed in `cco/realizable-holding`. The deterministic rule resolves it to `cco/mereotopology`.

4. **Is Part Of Process module corrected**: v3.2 placed in `cco/realizable-holding`. Rule resolves to `cco/mereotopology`.

5. **Part Of and Has Part removed**: v3.2 had `obo:BFO_0000050` (Part Of) and `obo:BFO_0000051` (Has Part). These IRIs DO NOT EXIST in BFO 2020 — they were generic parthood relations in older BFO releases that were removed/reorganized in BFO 2020. BFO 2020 instead has typed parthood (continuant_part_of, occurrent_part_of, plus their inverses). v3.3 carries the correct typed BFO 2020 parthood relations.

6. **Occupies Spatial Region IRI corrected**: v3.2 had `BFO_0000200` for "Occupies Spatial Region". In BFO 2020, `BFO_0000200` is actually "occupies spatiotemporal region" — a different relation. The correct IRI for spatial region occupation is `BFO_0000210`. v3.3 has both rows correctly: Occupies Spatial Region at BFO_0000210, and Occupies Spatiotemporal Region added at BFO_0000200.

7. **12 BFO 2020 ObjectProperties added**: v3.2 was missing has_temporal_part, material_basis_of, temporal_part_of, temporally_projects_onto, specifically_depended_on_by, occupies_spatial_region (corrected IRI), spatially_projects_onto, has_material_basis, first_instant_of, has_first_instant, last_instant_of, has_last_instant. v3.3 adds all of these from the loaded BFO 2020.

8. **OFI deontic content restored** with explicit [PROVENANCE-AUTHORED] markers acknowledging the non-query-grounded source.

9. **Has Role, Has Disposition, Has Function removed**: v3.2 had these as missing entries; v3.3-initial had them as [SME-DECISION-REQUIRED]. Label-search query confirmed they don't exist under any IRI in the loaded triplestore. Removed from v3.3 final. See section below for full rationale.

### Has Role / Has Disposition / Has Function — REMOVED

These three relations were authored in v3 with intended CCO IRIs but the IRIs do not exist in any loaded ontology (BFO 2020, CCO, IAO). A label-search query for "has role", "has disposition", "has function", and "patient in" returned zero results. They are not present in v0.1's source data under any IRI.

Per product owner decision, these rows have been removed from v3.3 entirely. If they are needed in v0.2 or later, they would need to be either:
- Sourced from an additional ontology not currently loaded (OBI, BFO-RO, etc.)
- Minted as OFI extensions with explicit subPropertyOf assertions
- Documented as conceptual relations the lifter handles via a different mechanism

For v0.1, they are out of scope.

### Four new modules (post-freeze ADR justified)

The spec §3.6.1 originally committed to 5 modules. v0.1 test corpus inspection revealed CCO content the existing modules don't cover. Justifications:

- **cco/measurement**: 8 relations covering measurement units, reference systems, time zones, geospatial coordinate reference systems. Required because v0.1 CCO Geospatial test corpus exercises is_geospatial_coordinate_reference_system_of and uses_geospatial_coordinate_reference_system.

- **cco/aggregate**: 5 relations covering collective bearer-of and inheres-in-aggregate vocabulary. Required because v0.1 Agent test corpus exercises aggregate bearer of, inheres in aggregate, capability of aggregate, aggregate has capability, aggregate has quality.

- **cco/organizational**: 14 relations covering affiliation, supervision, organizational context, subordinate roles, interest-bearing, delimitation. Required because v0.1 Agent test corpus exercises affiliation/supervision/organizational hierarchies extensively.

- **cco/deontic**: 8 relations covering CCO's own deontic vocabulary (prescribes, prohibits, permits, requires, plus their inverses). Distinct from ofi/deontic — CCO covers Directive→Action; OFI covers Directive→Issuing-Agent and the RDM v1.2.1 chain.

These four modules together resolve the issue that v3.2 had 29 rows in cco/realizable-holding that didn't actually fit the realizable-holding pattern. The deterministic rule now places those rows in modules that match their actual semantic role.

## Provenance markers in v3.3

Every row carries a provenance marker in column 13. Six marker types:

| Marker | Meaning | Action needed |
|---|---|---|
| `[QUERIED-CCO]` | Verified against loaded CCO via property-path rule | None — trust as authoritative |
| `[QUERIED-BFO]` | Verified against loaded BFO 2020 | None — trust as authoritative |
| `[QUERIED-IAO]` | Verified against loaded IAO | None — trust as authoritative |
| `[STABLE-TEXT]` | W3C standard vocabulary or FOL-only meta-level | None — these don't compile to ARC modules |
| `[DERIVED-LABEL]` | Module assigned by label semantics (no super-property in loaded data) | Aaron reviews and confirms or adjusts |
| `[SME-DECISION-REQUIRED]` | Genuine ambiguity; Aaron decides from spelled-out options | Aaron picks (a), (b), or (c) per row's Notes |
| `[PROVENANCE-AUTHORED]` | OFI deontic content; no source ontology to verify | Pending Reified Constitutive Relations Specification §3 publication |

A new provenance marker was introduced at this closeout: **[SME-CONFIRMED-LABEL]** (graduation pattern for [DERIVED-LABEL] rows that have been SME-reviewed and confirmed). This is bankable for AUTHORING_DISCIPLINE Phase 1 exit retrospective: when a row's module assignment is label-derived rather than query-grounded, the graduation pattern is `[DERIVED-LABEL] → [SME-CONFIRMED-LABEL]` (assignment confirmed) OR `[DERIVED-LABEL] → reassigned + [SME-CONFIRMED-LABEL]` (module changed) OR `[DERIVED-LABEL] → [QUERIED-CCO]` (later query surfaces a super-property declaration that wasn't in the original query). The marker discriminates the discipline applied: a query-grounded row is structurally verifiable from the upstream source; an SME-confirmed-label row is verifiable from the SME's reasoning recorded in the row's Notes column. Both are auditable; they document different verification disciplines.

## Trust attribution

Every row in v3.3 has a stated source. If a downstream consumer asks "where did this row come from?", the answer is in the Provenance column. This is the structural fix to the trust problem that motivated v3.3 generation.

This catalogue is genuinely produced from authoritative sources (for the BFO, CCO, and IAO content), with explicit acknowledgment of the rows that aren't (OFI authored, organizational/aggregate label-derived, IAO/CCO duplications pending SME decision). v3.2 had none of this provenance discipline; v3.3 has it for every row.
