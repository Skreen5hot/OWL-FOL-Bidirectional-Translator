# OFBT v0.2 Roadmap — Consolidated Closure Commitments

**Authored:** 2026-05-10 at Phase 3 close per Q-Frank-Step9-A Ask 6 (architect ruling 2026-05-10).
**Predecessor critique:** Frank's Phase 3 stakeholder critique [`demo/Phase3DemoCritique.md`](../demo/Phase3DemoCritique.md) §4.2 — flagged that "v0.2 ELK" was being invoked as the closure path for multiple gaps without scope/owner/timeline accountability; banking via consolidated artifact at the next phase boundary is the discipline.

---

## Purpose

This artifact consolidates every "v0.X closes Y" commitment from across Phases 1-3 of v0.1 into a single audit-trail-unified roadmap. Per the Q-Frank-Step9-A banked principle: **when deferred-closure-path framing accumulates across phases, a consolidated roadmap artifact at the next phase boundary lists every commitment with scope/owner/timeline.** Without consolidation, the deferred-closure framing becomes placeholder pattern that ships unfixed semantics without accountability.

The artifact's purpose is **accountability**, not commitment-creation. Each row records a commitment that has already been made in some Phase 1-3 artifact (demo, walkthrough, exit packet, ADR, fixture, ROADMAP); the roadmap aggregates them so a critic counting deferred items can see the full list with the load-bearing details.

---

## Conventions

- **Source phase / source gap:** the originating artifact + the specific gap/divergence it forward-tracked.
- **Closure mechanism:** what v0.2 (or v0.3+) is committed to do about it.
- **Scope (named features):** concrete features the closure mechanism delivers; not abstract intent.
- **Owner:** named person/role accountable for the closure landing. "TBD" if the project has not yet assigned one.
- **Timeline:** target phase or release. "TBD" if dependent on prior commitments not yet sequenced.

Honest acknowledgment per Frank's §4.2: many entries below carry "TBD" for owner and timeline. That is the v0.1 baseline — Phase 4-7 entry packets will progressively assign owners + timelines as the ARC-content workstream sequences. The artifact's discipline is to surface the TBDs explicitly rather than masking them under invocation phrases like "v0.2 ELK closes that."

## v0.2 closure commitments

### v0.2-01 — Layer A consistency-affirmation gap (load-bearing entry)

| Field | Value |
|---|---|
| **Source phase** | Phase 3 |
| **Source gap** | The v0.1 Horn-fragment classifier flags simple disjointness as `unverifiedAxioms` rather than affirming consistency on Horn-translatable subsets where no individual triggers the body. Per spec §8.5.1, simple disjointness IS Horn-checkable; the FOL translation `∀x. C(x) ∧ D(x) → False` is a Horn clause; satisfiability checks on individual named classes do not require consulting the disjointness axiom unless some individual is asserted to both classes. The BFO Layer A subset (`p1_bfo_clif_classical`) has no individual asserted to both Continuant and Occurrent, so the spec-required verdict is `'true'`; the implementation returns `'undetermined'`. |
| **Closure mechanism** | v0.2 ELK reasoner integration absorbs the Horn-fragment classifier refinement (distinguishing Horn-expressible-but-not-exercised from non-Horn-expressible) alongside ELK's broader Horn-fragment expansion. |
| **Scope** | (a) Horn-fragment classifier refinement: when an axiom is Horn-expressible but no individual triggers its body on the loaded ABox, the classifier marks it as Horn-Translatable-Inert rather than Horn-Fragment-Escape; (b) `checkConsistency()` returns `'true'` with `reason: 'consistent'` and empty `unverifiedAxioms` on subsets where all axioms are either Horn-Translatable-Active (no proof of inconsistency) or Horn-Translatable-Inert; (c) the Phase 4 demo's Layer A fragment (when re-introduced as Phase 4 demo's Case B per Phase 4 entry-packet authoring) demonstrates the closure. |
| **Owner** | TBD — assigned at v0.2 entry-cycle or Phase 4 entry packet ratification, whichever surfaces first. |
| **Timeline** | v0.2 release (post-Phase-7 v0.1 closeout). |
| **Cross-references** | `project/reviews/phase-3-step9-architectural-gap.md` Q-Frank-Step9-A Ask 1; `project/reviews/phase-3-exit.md` §7 Item 11; `tests/corpus/p1_bfo_clif_classical.fixture.js`; spec §8.5.1; spec §0.1 three-tier framing. |
| **Status** | Documented v0.1 implementation gap; Case B pulled from `demo/demo_p3.html` per Q-Frank-Step9-A Ask 1 (banked principle: demos do not claim what the implementation cannot demonstrate). |

## Update discipline

This artifact is updated at every phase boundary where a "v0.X closes that" commitment is added, modified, or closed. Updates land in the phase exit packet's commit alongside the artifact updates that produced the commitment.

When a commitment closes (the v0.X release ships the closure mechanism), the row migrates to a "Closed" subsection at the bottom of the artifact with the closing commit + verification context. The rolling list at the top stays focused on open commitments.

When a commitment is withdrawn (e.g., spec amendment removes the gap, OR the gap is closed earlier than projected), the row migrates to a "Withdrawn" subsection with the withdrawal cycle's architect ratification cross-reference.

Per Q-Frank-Step9-A banked principle: this artifact's purpose is accountability, not commitment-creation. Adding a row requires an existing commitment in some Phase 1-3 artifact; rows are not authored speculatively.
